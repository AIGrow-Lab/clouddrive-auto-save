# 优化由于缺少/错误提取码导致的网盘报错

## 目标 (Objective)
修复在需要提取码的网盘分享链接场景下，由于用户未填写或填写错误的提取码时，后端给出的错误提示过于生硬或晦涩的问题。提升错误提示的友好度并阻断后续的无用重试。

- **夸克网盘 (Quark)**: 将 `Quark token error: 41008` 映射为更易懂的 `[Fatal] 该分享链接需要提取码，但未提供或提取码错误`。
- **移动云盘 (139)**: 修复前端直接提示 `<nil>` 的问题。其根源在于 139 接口返回的错误码（如 `200000728`）在 Go JSON 反序列化后被解析为 `float64`，直接使用 `%v` 格式化时变成了科学计数法 `2.00000728e+08`，导致未能匹配到内置的错误映射表，最终 fallback 到了可能为空的 `message` 字段，从而输出了 `<nil>`。

## 具体改动 (Changes)

### 1. 修复 139 移动云盘错误码解析 (`internal/core/cloud139/client.go`)
在 `getShareInfo` 方法中：
- 优化对 `code` 字段的字符串转换。使用类型断言判断是否为 `float64`，如果是则使用 `strconv.FormatFloat(v, 'f', -1, 64)` 将其安全地转换为不带科学计数法的普通字符串。
- 将 `codeStr` 作为 Key 去匹配 `errorMap`，从而确保 `200000728` 可以正确地被映射为 `[Fatal] 提取码错误，请检查后再试。`
- 优化 fallback 逻辑：如果未匹配到映射表，且 `res["message"]` 为空时，回退输出为 `未知错误 (错误码: xxx)` 而不是 `<nil>`。

### 2. 优化夸克网盘错误提示 (`internal/core/quark/client.go`)
在 `getStoken` 方法中：
- 拦截特定错误码 `41008` (提取码为空或未提供)、`24000` (提取码错) 和 `41009` (其他 Token 错误)。
- 将这些错误码包装并抛出 `[Fatal] 当前分享链接需要提取码，未提供或提取码错误 (code: 41008)`。
- 添加 `[Fatal]` 前缀的目的是触发转存系统内置的“致命错误阻断”机制，将任务状态重置为 failed 且不进行死循环重试，提示用户前往修改。

## 验证方式 (Verification)
无需修改测试代码或运行完整 E2E，我们将直接审查代码逻辑。此后，使用缺失提取码的 139 或 Quark 链接进行测试，UI 会明确抛出提取码相关的中文错误。
